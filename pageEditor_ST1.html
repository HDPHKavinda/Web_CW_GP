Template & Navigation

Technical Description

template.html defines the structure every page in the site is built from: a <header> with the logo (an <img> with descriptive alt text) and a two-tier <nav> (utility bar plus main nav) with links to every page; a <main> left empty as the slot each student fills with their own content; and a <footer> with four columns (brand, page links, page-owner links, student details). The colour palette went through real iteration before landing here: since SDG 4 is officially represented with a red icon, our first attempt used red as the site's primary colour, but once it was actually applied across a full page it read as harsh and closer to the colour of blood than anything associated with education, so the group met and agreed to move to a blue/yellow-accent palette instead, which is what led to the "BrightPath" name. The nav bar itself also went through several attempts: an early version had the SDG/accessibility text as a dark bar directly under the main nav, which didn't read well, so it was moved above the main nav instead as the current two-tier utility-bar-plus-nav structure, inspired in part by the centred-logo nav layout on sleepwiseconsulting.com (reference 8) — though the final design, colours, and content are original, not copied. The header uses position: sticky, with JavaScript (js/navbar.js) adding a header-scrolled class once the user scrolls past a threshold to animate the utility bar closed; this specifically exists because on mobile the full header was still taking up too much vertical space even after switching to a hamburger menu for the links, so collapsing the utility bar away on scroll was the fix that actually solved it. On narrow screens the hamburger toggle button was arrived at after researching common mobile-nav patterns, using CSS order values so the logo and toggle stay together whether the menu is open or closed. The shared colour palette and the Poppins/Inter font pairing (reference 6) are defined centrally in css/global.css so every page inherits them automatically. The logo itself (reference 5) was AI-generated by Student 2 rather than sourced from a stock site.

Accessibility

All images, including the logo, have descriptive alt text. The header, nav, main, and footer use semantic HTML elements rather than generic divs, and the mobile toggle button carries aria-label and aria-expanded, kept in sync with its open/closed state by JavaScript.

Home page

Technical Description

home.html follows the shared template, then adds a full-bleed hero banner (background photo credited in reference 1) using the width: 100vw; position: relative; left: 50%; margin-left: -50vw; technique to break out of the centred container. Finding a hero photo that actually matched the finished blue colour palette took several attempts before settling on the current one. The card section went through a real design problem: the spec asks for at least 3 mission cards, but the site also needed to link to exactly 4 students' Content Pages, and a single card grid can't cleanly represent both a "3" and a "4" at once — after several layouts that tried to force both into the same grid, the fix was to split them into two separate, purpose-built sections instead: a CSS Grid of three content cards (repeat(auto-fit, minmax(280px, 1fr))) each with a photo (credited in reference 2 and reference 3), heading, description and checklist, and a separate focus-list of exactly four links to all four students' Content Pages. The footer design was chosen after comparing roughly ten real websites' footers and reading through an education-site design roundup (reference 9) for general layout ideas, before designing an original four-column footer specific to this site. Page-specific CSS (the hero fade-in animation, card hover-lift, focus-list hover states) is embedded in a <style> block, as required for this page.

Accessibility

The hero photo and all three card images have descriptive alt text rather than filenames. The focus-list links to each teammate's Content Page are real <a> elements, keyboard reachable by default. The validator originally flagged the mission-card headings for jumping from <h1> straight to <h3>; adding a proper <h2> before them fixed the outline for screen-reader navigation.

Gallery Page

Technical Description

Each thumbnail in gallery.html is a <div class="thumb" tabindex="0" role="button"> carrying data-title, data-desc, and data-img attributes. The hover feedback and click-to-open behaviour both went through a couple of redesigns before landing on the current version (reference 10) — an earlier attempt at how the extended view opened didn't feel right, so the interaction was reworked to open more predictably and be easier to close again. A single hidden modal is populated by JavaScript rather than duplicated per photo. All interactivity (opening/closing the modal, hover feedback, and the colour/font toggles inside the modal) is handled with addEventListener (reference 7) rather than CSS :hover — the spec explicitly states that submissions relying solely on CSS techniques "will not meet this requirement," so every interaction here is deliberately JS-driven for that reason. Page-specific styling and script live in their own external files, css/gallery.css and js/gallery.js. The four thumbnail photos are credited in reference 2, 3 and 12.

Accessibility

Each thumbnail carries tabindex="0", role="button", and a descriptive aria-label, since a plain <div> isn't natively focusable or announced as interactive. The keydown handler responds to both Enter and Space. The modal's close button has its own aria-label="Close", and the modal can also be dismissed with the Escape key.

Content Page

Technical Description

content_ST1.html has one <h1> and five <h2> sections, each with a unique id. The internal navigation went through a design change: the first version listed the section links horizontally, styled like a second nav bar, but after looking at how other sites handle long-article navigation it was rebuilt as a vertical sidebar list instead, since that made every link easier to see and click without competing for the same horizontal space as the page's own title. That sidebar is built with CSS Grid (grid-template-columns: 230px 1fr), with position: sticky on the sidebar so it stays in view while the article scrolls. A media query collapses this to one column below 820px. The "Go to Top" button is a plain anchor link styled with position: fixed — deliberately HTML/CSS only, as required. Three of its four images are credited in reference 4; the fourth (the vocational workshop photo, also used on the Gallery Page) is credited in reference 12.

Accessibility

The page keeps a single <h1> followed by five <h2> section headings with no skipped levels. The sidebar links are ordinary anchor links, and each target section has scroll-margin-top so jumping to it doesn't hide the heading under the sticky site header. All four article images have descriptive alt text.

Challenges and Lessons Learned

Getting the colour palette right took longer than I expected. SDG 4's official icon is red, so the very first version of the site used red as the main colour — but once I actually built a full page with it, it looked less like education and more like blood on screen. I brought it up in a group meeting, we agreed it wasn't working, and Nidarsha rebuilt the palette around blue with a yellow/sky-blue accent instead, which is also where the name "BrightPath" came from.

The nav bar went through a few real redesigns too. I liked the centred-logo layout on sleepwiseconsulting.com (reference 8) and used it as a starting point, though the final colours, content, and structure here are our own, not copied. My first attempt at the sub-nav put the "SDG 4 · Quality Education" text as a dark line directly under the main nav — it looked cramped, so I moved it above the main nav instead as its own thin bar (.utility-bar), which became the two-tier header the site actually uses now.

Mobile was its own problem. The first mobile nav just stacked every link open by default, fixed in place, and it ate up way too much of the screen. I looked into how other sites handle this and landed on the hamburger-menu pattern (.nav-toggle, toggled with classList.toggle('nav-open') in js/navbar.js) — but even with the menu collapsed behind a button, the utility bar above it was still taking up space permanently. The fix was making the header react to scrolling: at the top of the page you see the full two-row header, but scroll down and JavaScript adds a header-scrolled class that animates the utility bar away, leaving just the compact nav row pinned to the top.

On the Home page, the real problem was that I had 3 mission cards but 4 students' Content Pages to link to, and one grid can't represent both a "3" and a "4" cleanly. I tried a few layouts that tried to force both into the same card grid before giving up and splitting them into two separate sections instead — a 3-card .cards-grid for the mission content, and a separate 4-item .focus-list purely for linking to everyone's Content Page. The footer took a lot of comparison too — I looked at close to ten real site footers and an education-site design roundup (reference 9) before designing this one from scratch.

The Gallery went through its own redesign. My first version of the extended-view popup didn't open or close in a way that felt right, so I rebuilt the interaction — the .thumb elements, the shared modal, and all the open/close/hover logic in js/gallery.js — until it worked the way it does now (reference 10). Everything had to be handled with addEventListener rather than CSS :hover, since the spec is explicit that a CSS-only gallery "will not meet this requirement."

Even the Content Page's sidebar changed shape. I first laid the section links out horizontally, like a second nav bar, but it competed for space with the actual page title and felt cramped, so I rebuilt it as the vertical, sticky sidebar it is now (position: sticky; top: 100px;) — which matches how most long-form articles handle in-page navigation anyway. General page-layout browsing during this stage, including sites like the two in reference 11, also fed into how I thought about laying content out, even where I can't point to one exact feature I borrowed from them.

Coordinating file names and the shared style.css with teammates so nobody's page broke another's was an ongoing thing too — a small naming mismatch in one shared file could silently break someone else's page.

Accessibility Considerations

Across the template, Home, Gallery and Content pages, accessibility was built in rather than added afterwards. The shared <header>, <nav>, <main> and <footer> use real semantic elements instead of generic <div>s, so the page's structure is announced correctly to screen readers before any content is even read. Every image across all four pages, including the logo, carries descriptive alt text rather than a filename or empty string. The mobile nav toggle keeps its aria-expanded attribute in sync with its open/closed state via JavaScript, so assistive technology always reports the current menu state correctly. On the Gallery page, thumbnails are plain <div>s made keyboard-operable with tabindex="0", role="button" and a descriptive aria-label, since a div isn't natively focusable or announced as interactive; the modal responds to both Enter and Space, has its own aria-label="Close" on the close control, and can also be dismissed with Escape. On the Content page, heading levels never skip (a single <h1> followed by properly nested <h2>s), and every anchor-linked section has scroll-margin-top so jumping to it doesn't hide the heading behind the sticky site header. These choices matter because they mean the site works the same way whether someone is using a mouse, a keyboard, or a screen reader, rather than assuming everyone interacts with it the same way I do.
